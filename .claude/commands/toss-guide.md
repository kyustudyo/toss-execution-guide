# Toss Guide

사용자가 어떤 기능, 정책, SDK, 구조, 제품 요구사항을 주더라도 아래 기준으로 작업합니다.

## 목표
- 결과물은 단일 HTML 파일입니다.
- 문서는 토스 스타일의 높은 가독성을 가져야 합니다.
- 읽는 사람이 무엇을 해야 하는지 바로 이해할 수 있어야 합니다.

## 반드시 포함할 내용
- 한 줄 요약
- 왜 하는가
- 핵심 포인트
- 범위 / 비범위
- AS-IS / TO-BE
- 실행 계획
- 데이터 / API / 설정 계약
- 클래스 다이어그램
- 시퀀스 다이어그램
- 플로우차트
- 구현 체크리스트
- QA 체크리스트
- 운영 체크리스트
- 정책 / 리스크 / 남은 작업

## 조사 원칙
- 제공된 요구사항과 로컬 코드를 우선 확인합니다.
- 확인된 사실과 추정, 미확정 사항을 분리합니다.
- 원문을 복붙하지 않고 실행 관점으로 재구성합니다.

## 출력 규칙
- 기본 파일명은 `<TOPIC>_IMPLEMENTATION.html`
- 필요하면 Markdown 소스도 같이 생성
- 다이어그램은 실제 연결선이 보이도록 작성
- 필요하면 Mermaid 사용 가능
- 큰 다이어그램은 `+` 버튼과 한 화면 맞춤 팝업 허용
